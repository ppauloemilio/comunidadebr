1. Mapeamento de Entidades (Schema Snapshot)
Aqui estão as entidades que compõem o seu ecossistema de dados atual:

Social & Conteúdo:
Post: Gerencia o feed, tipos (texto, imagem, promo, evento), metadados e contador de interações.
Comment & Like: Interações sociais sobre posts e comentários.
Share: Compartilhamento de conteúdo.
Follow: Conexões entre usuários.
Friendship: Solicitações e vínculos de amizade.
Negócios & Localização:
Business: Perfil completo da empresa (proprietário, endereço geolocalizado, links sociais, fotos).
Country, State, City: Estrutura de localização geográfica do app.
Perfil & Usuário:
PublicProfile: Dados públicos do usuário (bio, redes, fotos, moradia).
UserCountryHistory: Histórico de moradia do usuário para filtrar o feed.
Skill & UserSkill: Habilidades profissionais dos usuários.
Comunicação:
Conversation: Gerencia conversas (User-User ou User-Business).
Message: Histórico de mensagens, anexos e status de leitura/deleção.
Notification: Sistema de alertas do sistema.

O sistema baseia-se em 18 entidades principais. O foco da migração deve ser a replicação da estrutura JSON e das permissões de leitura/escrita.

Publicação e Engajamento: Post, Comment, Like, Share, Notification.
Negócios: Business, Advertisement.
Perfil e Social: PublicProfile, Friendship, Follow, UserSkill, Skill.

Comunicação: Conversation, Message.

2. Mapa de Rotas e UI (Arquitetura)
A aplicação é construída sobre um roteador único com LayoutWrapper global.

Página Principal: /Feed (Landing page).
Estrutura Global: Header (Desktop/Mobile), Bottom Navigation (Mobile), e Overlays (Mobile Menu).
Tecnologias de Interface: React (Vite), Tailwind CSS, shadcn/ui (primitivas Radix), lucide-react (ícones), TanStack Query (cache de estado).
Notas para Migração: Ao mudar de ambiente, certifique-se de replicar o Layout.jsx, pois ele centraliza as chamadas de API para notificações (Notification) e conversas (Conversation), que alimentam as badges de alerta em toda a interface.
3. Regras de Negócio e Fluxos (Lógica Core)
A. O "Geofiltro" (Regra de Ouro)
O Feed.jsx contém a lógica mais importante do sistema. Para replicar o comportamento atual, o novo backend/frontend deve:

Consultar o histórico do usuário (UserCountryHistory).
Comparar post.created_date com o joined_at do histórico do país do autor.
Ocultar posts de membros do mesmo país que tenham sido criados antes da entrada do usuário atual na comunidade local.
B. Ciclo de Vida de Dados
Soft-Delete: NUNCA deletar fisicamente registros de Business ou Post. Utilize a flag is_active: false.
Snapshot de Dados: A entidade Share salva um snapshot do post original. Isso é fundamental para manter a integridade visual se o post original for deletado ou editado.
Desnormalização: Para manter a performance, as entidades de engajamento (Like, Follow, Comment) replicam dados básicos do autor (name, photo, username). Isso evita a necessidade de realizar "JOINs" lentos entre o usuário e o post.
C. Comunicação
O sistema de chat funciona em dois tipos: user_user e user_business. A entidade Conversation armazena o estado das mensagens não lidas (unread_count) via um objeto, facilitando a contagem por participante no frontend.

A "Etapa 3: Regras de Negócio e Fluxos Críticos" define como o sistema "pensa" e executa as ações mais importantes. Abaixo, o mapeamento dos fluxos que garantem o comportamento social e geográfico do Comunidade Brasil V2.

1. Motor de Filtragem Geográfica (O diferencial social)
A regra mais crítica do sistema está no componente Feed.jsx (linhas 62-92). Ela não é apenas uma lista de posts; é um filtro inteligente:

Lógica de visibilidade: O sistema cruza o país atual do usuário com o UserCountryHistory.
Data de Entrada: O usuário não vê posts antigos de um país que ele acabou de entrar. Ele só recebe conteúdo postado após a data (joined_at) registrada no histórico.
Permissividade: Se o usuário nunca registrou uma data de entrada (ou mora lá há muito tempo), ele visualiza todo o histórico de posts daquela comunidade.
2. Fluxo de Interações Sociais (Engagement Loop)
O sistema gerencia o engajamento através de mutações no React Query, garantindo consistência:

Like/Unlike:
O sistema verifica se o postId já existe no userLikes (Set de IDs).
Ao curtir: Cria a entidade Like com snapshot dos dados do usuário (nome, foto, username) e incrementa o likes_count diretamente na entidade Post.
Ao descurtir: Remove a entidade Like e decrementa o contador, garantindo a integridade dos dados (Math.max(0, count - 1)).
Realtime/Refresh: O uso de queryClient.invalidateQueries garante que o feed seja re-renderizado imediatamente após qualquer interação do usuário.
3. Gestão de Entidades (Ciclo de Vida)
Soft-Delete (Negócios e Posts): O sistema utiliza a flag is_active: true para "excluir" registros. Isso é crítico para manter o histórico de comentários e likes de um post ou empresa sem deletar fisicamente o registro do banco de dados.
Desnormalização de Dados: O sistema armazena dados de autores e usuários em diversas entidades (Post, Comment, Like, Follow). Isso permite leitura rápida sem necessidade de "joins" complexos no frontend, otimizando o carregamento do feed.
4. Fluxo de Notificações
Sempre que uma interação ocorre (ex: alguém curte um post ou segue um usuário), a entidade Notification é criada com:

actor_id (quem fez).
target_type e target_id (o que foi afetado).
A leitura da notificação dispara um is_read: true, que remove o indicador de contagem no Layout.jsx.
